# BarByBar 单人版 Web 化改造方案

## 1. 文档目的

本文档用于评估并规划将当前 `BarByBar` Windows 桌面程序改造成“单人自用的 Web 程序”的可行路径。

当前阶段只做方案设计，不进入开发。

本文档重点回答以下问题：

- 现有项目哪些部分可以复用，哪些必须重做
- 单人版 Web 化的合理目标边界是什么
- 采用什么技术方案最稳妥
- 应该如何分阶段实施，避免一次性重写失败
- 部署到服务器时要关注哪些工程问题

## 2. 当前项目现状

### 2.1 当前产品形态

当前项目是一个 Windows 桌面复盘工具，主要特征如下：

- 使用 Python 3.11
- UI 技术栈为 `PySide6 + pyqtgraph`
- 数据来源为本地 CSV
- 数据库存储为本地 SQLite
- 打包方式为 `PyInstaller`
- 当前发布目标是 Windows 可执行程序

### 2.2 当前主要能力

从现有代码和 README 可确认，系统已具备以下核心能力：

- 导入分钟级 OHLCV CSV 数据
- 创建回放会话
- 按 K 线逐步前进、后退、跳转
- 记录开仓、加减仓、平仓、备注
- 设置止损止盈线和其他订单线
- 保存会话、动作、统计结果
- 回看历史复盘结果
- 绘制图表标注和辅助线

### 2.3 当前架构判断

现有代码可以粗分为三层：

1. 领域层
   - `src/barbybar/domain`
   - 包含交易回放引擎、模型定义、状态流转

2. 存储与数据层
   - `src/barbybar/storage`
   - `src/barbybar/data`
   - 包含 CSV 导入、时间周期转换、SQLite 持久化

3. 桌面 UI 层
   - `src/barbybar/ui`
   - `src/barbybar/app.py`
   - 包含 Qt 主窗口、图表组件、事件处理和交互逻辑

这个分层对 Web 化是有利的，因为核心业务逻辑并未完全耦合在 GUI 里。

## 3. 结论摘要

## 3.1 总体可行性

将本项目改造成“单人自用 Web 程序”是可行的，而且可行性较高。

但需要明确：

- 这不是“直接部署现有程序”
- 这也不是“把 PySide6 界面转成网页”
- 本质上是“复用 Python 核心逻辑，重写前端交互层”

### 3.2 复用判断

高复用部分：

- `domain` 下的业务模型与回放引擎
- `data` 下的 CSV 导入、时间周期聚合等逻辑
- `storage` 下的仓储思路和数据库结构
- 大部分现有单元测试思路

低复用或不可复用部分：

- `PySide6` 主窗口代码
- `pyqtgraph` 图表渲染代码
- Qt 线程、信号槽、对话框、文件选择器
- 桌面交互细节

### 3.3 推荐目标

建议第一阶段只做：

- 单用户
- 自托管
- 浏览器访问
- 不做注册登录
- 不做多人并发
- 不做复杂权限隔离
- 不做正式商业化部署能力

这是当前最稳妥、投入产出比最高的目标。

## 4. 单人版 Web 化的目标边界

为了避免方案失控，建议先明确“做什么”和“不做什么”。

### 4.1 本期建议纳入

- 浏览器上传 CSV
- 数据集管理
- 创建复盘会话
- 查看 K 线图
- 逐 bar 前进
- 回退到历史状态
- 跳转到目标位置
- 记录交易动作
- 设置止损止盈与基础订单线
- 编辑笔记与标签
- 查看统计结果
- 保存与恢复会话
- 在单台服务器或家用主机上部署

### 4.2 本期建议暂不纳入

- 多用户账户系统
- 用户权限模型
- 团队共享数据
- 云对象存储
- 实时行情接入
- 分布式任务队列
- WebSocket 实时协作
- 完整移动端适配
- 与桌面版完全一致的所有绘图工具
- 复杂批量导入与数据清洗后台任务

### 4.3 为什么要先收缩范围

当前系统最复杂的部分不是“算逻辑”，而是“交互体验”。尤其是：

- 图表交互
- 订单线拖拽
- 标注绘制
- 局部窗口加载与回放同步

如果一开始就追求“桌面版全部能力等价迁移”，工期和复杂度会显著上升。

## 5. 推荐技术路线

## 5.1 总体路线

推荐采用前后端分离，但保持技术栈尽量简单：

- 后端：Python + FastAPI
- 前端：React
- 数据库：SQLite（单人版可接受）
- 部署：Docker 或直接 systemd/pm2 风格进程托管
- 反向代理：Nginx

这是最符合当前项目实际情况的方案。

## 5.2 为什么推荐 FastAPI

原因如下：

- 现有核心逻辑已经在 Python 中
- 可以直接复用 `domain/data/storage` 代码
- API 层封装清晰，适合逐步替换桌面入口
- 学习成本和迁移成本都低
- 对单人版部署足够轻量

### 5.3 为什么前端建议用 React

原因如下：

- 图表与复杂交互更适合浏览器前端处理
- 可以较自然地管理页面状态和会话状态
- 生态里有成熟的图表库和拖拽交互方案
- 后续如果要扩展多人版，也更容易演进

如果你更偏好极简，也可以考虑：

- 后端模板渲染 + 少量 Alpine.js/HTMX

但对这个项目来说，这条路不适合长期发展，因为图表交互较复杂。

## 5.4 图表技术建议

图表是 Web 化成功与否的关键。

推荐优先评估：

1. TradingView Lightweight Charts
2. Apache ECharts
3. Plotly

建议排序：

1. `Lightweight Charts`
   - 更贴近交易图表场景
   - 性能较好
   - K 线体验更自然
   - 但高级绘图能力需要自行补很多交互

2. `ECharts`
   - 通用能力强
   - 扩展性尚可
   - 自定义交互也可以做
   - 但交易场景手感通常不如专用库

3. `Plotly`
   - 上手快
   - 用于快速原型合适
   - 但不适合长期承载复杂交易式交互

### 5.5 对图表交互的现实判断

桌面版中以下能力迁移成本最高：

- 绘图工具
- 图元命中测试
- 拖拽修改锚点
- 十字光标联动
- 订单线拖动与吸附
- 局部窗口滚动加载

因此建议第一期只保留：

- K 线显示
- 当前 bar 高亮
- 简单订单线显示
- 买卖点标记
- 基础笔记信息

绘图系统建议第二期再做。

## 6. 推荐的新架构

建议将项目逐步调整为如下结构：

```text
BarByBar/
  docs/
  src/
    barbybar/
      domain/
      data/
      storage/
      services/
      api/
      webapp/        # 如果前端静态资源由后端托管，可保留此目录
      ui/            # 桌面版暂时保留，直到确认废弃
  frontend/          # React 前端
  tests/
```

### 6.1 各层职责

`domain`

- 纯业务逻辑
- 不依赖 Web 框架和 GUI
- 保持可测试性

`storage`

- 仓储与数据库访问
- 未来可以抽象为 SQLite/PostgreSQL 双实现

`services`

- 编排业务流程
- 负责 API 层调用领域对象
- 处理事务边界与 DTO 转换

`api`

- FastAPI 路由
- 请求验证
- 响应序列化

`frontend`

- 页面、组件、交互逻辑、图表渲染

### 6.2 一个关键原则

从现在开始，应该把“桌面 UI”和“业务逻辑”彻底分离。

未来不论是否继续保留桌面版，核心逻辑都应该只在一处维护。

## 7. 单人版部署方案

## 7.1 推荐部署模式

对于单人自用，推荐下面两种部署模式之一。

### 模式 A：单机自托管

适合：

- 你自己的云服务器
- 家里小主机
- NAS

组成：

- FastAPI 后端
- React 前端静态资源
- SQLite 数据库文件
- 本地上传目录
- Nginx 反向代理

优点：

- 简单
- 成本低
- 运维负担轻

缺点：

- SQLite 并发能力有限
- 后续扩展多人版时需要迁移数据库

### 模式 B：Docker 部署

组成：

- `app` 容器
- `nginx` 容器
- 挂载数据卷保存数据库和上传文件

优点：

- 部署可复制
- 迁移方便
- 本地和服务器环境更一致

缺点：

- 比直接跑 Python 稍复杂

对于你这种单人版场景，我更建议：

- 如果你本身熟悉 Docker，就用 Docker
- 如果你偏向稳定省心，就用 `FastAPI + Nginx + systemd`

## 7.2 数据库存储建议

单人版继续使用 SQLite 是合理的，但要调整存储策略。

当前桌面版默认把数据库放在用户家目录下。Web 版建议改为：

- 明确的服务器数据目录
- 例如 `/opt/barbybar/data/` 或 `/srv/barbybar/data/`

建议存储拆分为：

- `app.db`：SQLite 数据库
- `uploads/`：上传 CSV 原文件
- `logs/`：运行日志
- `exports/`：将来如需导出可预留

## 7.3 备份建议

单人版也要有备份策略，否则数据积累后风险很高。

建议至少具备：

- 每日自动备份 SQLite 文件
- 每日备份上传目录
- 保留最近 7 到 30 天

如果部署在云服务器上，最好额外同步到对象存储或网盘。

## 8. 详细改造思路

## 8.1 第一步：先抽出真正可复用的核心

目标：

- 在不改业务行为的前提下，把核心逻辑从桌面入口彻底独立出来

具体动作：

- 检查 `domain` 是否引用任何 UI 类型
- 检查 `storage` 是否混入界面状态假设
- 明确会话状态、动作、订单线、绘图数据的 API 表达形式
- 让核心逻辑可以在命令行测试中独立运行

这一步是整个改造成败的基础。

## 8.2 第二步：为核心逻辑包一层服务层

建议新增 `services` 层，提供明确的用例接口，例如：

- `import_dataset`
- `list_datasets`
- `create_session`
- `load_session`
- `step_forward`
- `step_back`
- `jump_to`
- `record_action`
- `place_order_line`
- `save_notes`
- `save_tags`
- `get_session_stats`

这样做的好处是：

- API 层不直接操作 Repository 和 Engine 细节
- 将来桌面版和 Web 版都能复用同一套服务

## 8.3 第三步：定义 REST API

建议第一期 API 范围如下：

数据集：

- `POST /api/datasets/import`
- `GET /api/datasets`
- `GET /api/datasets/{id}`
- `DELETE /api/datasets/{id}`

会话：

- `POST /api/sessions`
- `GET /api/sessions`
- `GET /api/sessions/{id}`
- `POST /api/sessions/{id}/step-forward`
- `POST /api/sessions/{id}/step-back`
- `POST /api/sessions/{id}/jump`
- `PATCH /api/sessions/{id}/notes`
- `PATCH /api/sessions/{id}/tags`

动作与订单：

- `POST /api/sessions/{id}/actions`
- `POST /api/sessions/{id}/order-lines`
- `DELETE /api/sessions/{id}/order-lines/{line_id}`

图表与窗口：

- `GET /api/sessions/{id}/chart-window`

绘图：

- 第一阶段可先不开放或只做只读

## 8.4 第四步：重建前端界面

前端页面建议最少包括：

- 登录页：第一期可以没有
- 数据集列表页
- 新建导入页
- 会话列表页
- 回放主页面
- 统计/历史查看页

回放主页面应优先包含：

- 左侧或顶部基础控制区
- 中间 K 线图
- 右侧动作面板
- 底部或侧边笔记/标签区域
- 状态栏显示当前位置、仓位、PnL、订单线等

## 8.5 第五步：迁移文件导入方式

桌面版依赖本地文件选择器。Web 版要替换为：

- 浏览器上传 CSV
- 后端保存到上传目录
- 后端解析并入库

要增加的校验包括：

- 文件大小限制
- 文件类型限制
- 列映射校验
- 日期解析失败提示
- 重复导入策略

## 8.6 第六步：调整持久化边界

当前是桌面单机思路，Web 版需要更明确地处理：

- 请求之间的会话状态一致性
- 页面刷新后的恢复
- 未保存状态如何处理
- 自动保存频率

建议第一期采用：

- 所有关键操作立即持久化
- 不依赖浏览器本地缓存维持业务正确性

这样实现更稳。

## 9. 分阶段实施建议

## 9.1 Phase 0：方案确认

输出目标：

- 确认是否只做单人版
- 确认是否保留桌面版并行存在
- 确认前后端技术栈
- 确认第一期必须功能清单

交付物：

- 本文档

## 9.2 Phase 1：后端底座

目标：

- 不碰前端，先把 Python 核心能力 API 化

工作项：

- 引入 FastAPI
- 抽 `services`
- 新增 Pydantic 模型
- 定义基础 API
- 跑通 CSV 导入、创建会话、逐 bar 前进
- 增补接口测试

完成标志：

- 不打开桌面 UI，也能通过 API 驱动一个完整回放流程

## 9.3 Phase 2：最小可用 Web 前端

目标：

- 有一个能用的浏览器版本

工作项：

- 新建 React 前端
- 完成数据集导入页面
- 完成会话管理页面
- 完成基础 K 线回放页面
- 完成动作记录与统计展示

完成标志：

- 可以在浏览器里完成基本复盘闭环

## 9.4 Phase 3：交互增强

目标：

- 提升接近桌面版的体验

工作项：

- 图表缩放和平移
- 订单线交互增强
- 图表窗口加载优化
- 局部缓存优化
- 基础绘图工具

## 9.5 Phase 4：部署与稳定性

目标：

- 形成可长期自用的服务

工作项：

- Docker 化或 systemd 化
- Nginx 配置
- HTTPS
- 备份脚本
- 日志轮转
- 崩溃恢复

## 10. 关键技术决策

## 10.1 是否继续使用 SQLite

结论：

- 单人版可以继续使用 SQLite
- 第一阶段不必急着换 PostgreSQL

前提：

- 同时只有你自己使用
- 不做高并发
- 不做复杂后台并行任务

如果后续出现以下情况，再考虑迁移：

- 多用户
- 多终端同时操作
- 频繁大文件导入
- 大量统计查询

## 10.2 是否保留桌面版

建议：

- 第一阶段保留桌面版，不立即删除

原因：

- 桌面版是现成可用系统
- Web 版初期很难立刻达到同等交互体验
- 保留桌面版可以作为行为对照基线

更现实的策略是：

- 先做 Web MVP
- 验证一段时间
- 再决定是否废弃桌面版

## 10.3 是否一步迁移全部绘图功能

不建议。

建议顺序：

1. 先完成交易回放闭环
2. 再做订单线
3. 最后做复杂绘图工具

这样最稳。

## 10.4 是否做认证登录

如果只是你自己用，建议第一阶段不做完整用户系统。

可选方案：

- 仅内网访问
- Nginx Basic Auth
- 简单单账号口令保护

不要一开始就引入完整 RBAC。

## 11. 风险清单

## 11.1 最大风险：图表交互体验下滑

桌面图表交互非常容易比 Web 版更丝滑。

风险表现：

- 拖拽不顺手
- 命中区域难调
- 缩放和平移体验差
- 绘图工具难用

应对建议：

- 第一阶段降低图表交互目标
- 先验证回放逻辑闭环
- 用真实复盘流程测试，而不是只看页面截图

## 11.2 第二风险：领域逻辑与 UI 隐式耦合

虽然当前分层还不错，但实际改造时可能发现：

- 某些状态只在 UI 中维护
- 某些默认行为依赖界面事件顺序
- 某些临时逻辑没有明确的服务边界

应对建议：

- 先写 API 级集成测试
- 以现有测试为基础补充关键流程测试

## 11.3 第三风险：SQLite 使用方式不当

单人版能用 SQLite，但依然要注意：

- 长事务
- 同时导入和操作会话
- 数据库锁竞争
- 文件备份时机

应对建议：

- 保持事务短小
- 导入与交互分离
- 明确备份窗口

## 11.4 第四风险：范围蔓延

最常见失败方式不是技术难，而是目标不断增加，例如：

- 顺便做用户系统
- 顺便做云同步
- 顺便做移动端
- 顺便做策略分析

应对建议：

- 任何新需求先放到“第二阶段以后”
- 第一阶段只看能否稳定完成单人复盘

## 12. 工作量预估

以下为粗略估算，不作为承诺工期，只用于决策。

### 12.1 如果目标是最小可用版本

范围：

- CSV 导入
- 会话管理
- 基础回放
- 动作记录
- 简单统计
- 基础图表

预估：

- 1 名熟悉 Python + React 的开发者
- 大约 2 到 4 周

### 12.2 如果目标是接近桌面版体验

新增：

- 更完整的订单线交互
- 更复杂的图表行为
- 绘图工具
- 更好的状态恢复

预估：

- 大约 4 到 8 周以上

### 12.3 为什么差距这么大

因为真正耗时的是“交互系统”，不是“普通 CRUD”。

## 13. 推荐的最小决策集

在真正进入开发前，建议你只先拍板下面几个问题：

1. 是否确认只做单人版
2. 是否确认保留桌面版一段时间
3. 是否接受第一期不完整迁移绘图能力
4. 是否接受前端重写而不是直接套桌面界面
5. 是否采用 `FastAPI + React + SQLite`
6. 是否以 Docker 或 Nginx + systemd 为部署目标

只要这几个问题定下来，方案就可以进入实施设计阶段。
