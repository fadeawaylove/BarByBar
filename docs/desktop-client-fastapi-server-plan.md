# BarByBar 桌面客户端 + FastAPI 服务端分离方案

## 1. 文档目的

本文档用于规划 `BarByBar` 从“本地单机桌面程序”演进为“桌面客户端 + 远程服务端”的分离式架构。

该方案的核心目标不是 Web 化，而是：

- 保留现有桌面客户端体验
- 将数据和核心持久化能力集中部署到服务器
- 让你在不同电脑上运行客户端时，访问同一份个人数据

当前阶段只做方案设计，不进入开发。

## 2. 目标定义

## 2.1 要解决的问题

你当前真正关心的是：

- 不同电脑都能使用同一份复盘数据
- 不需要把整个系统重写成浏览器网页
- 尽量保留现有桌面端图表和交互体验
- 后端统一部署到服务器，便于长期使用和备份

这个目标非常适合采用“客户端和服务端分离”的路线。

## 2.2 本方案的核心结论

该方案可行，而且相比 Web 重写更适合当前项目。

你不必把 `PySide6` 客户端废掉。更现实的路径是：

1. 保留桌面客户端
2. 引入 `FastAPI` 作为服务端
3. 将数据存储、会话持久化、CSV 导入、统计查询等能力迁移到服务器
4. 客户端通过 HTTP API 调用服务端

这样可以实现：

- 任意电脑上打开客户端，只要能连上服务器，就能读取同一份数据
- 未来如果要做 Web 版，也能在这个服务端基础上继续扩展

## 3. 当前架构适配性分析

## 3.1 当前代码结构的优点

现有项目并不是完全“UI 和逻辑糊在一起”的结构，而是已经有较明显分层：

- `domain/`
  - 回放引擎
  - 交易动作逻辑
  - 仓位、订单线、统计模型

- `data/`
  - CSV 导入
  - 时间周期聚合
  - tick size 处理

- `storage/`
  - SQLite schema
  - Repository

- `ui/`
  - Qt 主窗口
  - 图表组件
  - 桌面交互逻辑

这意味着：

- 图形界面可以继续保留
- 核心逻辑可以逐步服务化
- 风险远小于直接改成 Web 前后端重写

## 3.2 当前不适合直接共享的部分

当前项目仍然是“本地直连数据库”的桌面思路，典型表现如下：

- `Repository` 直接连接 SQLite
- 默认数据库路径在本地用户目录
- UI 直接实例化仓储对象并操作数据
- CSV 导入依赖本地文件路径
- 会话生命周期默认由本地进程掌控

这些部分不适合多设备共享，需要调整为“服务端统一管理”。

## 4. 推荐的新架构

## 4.1 总体架构

推荐架构如下：

```text
+---------------------+        HTTPS / HTTP API        +----------------------+
| Desktop Client A    |  --------------------------->  | FastAPI Server       |
| PySide6 + Charts    |                                 | Application Service  |
+---------------------+                                 | Domain / Repository  |
                                                        | Database             |
 +---------------------+                                +----------------------+
 | Desktop Client B    |  --------------------------->         |
 | PySide6 + Charts    |                                        |
 +---------------------+                                        v
                                                        +----------------------+
                                                        | PostgreSQL / SQLite  |
                                                        | CSV Upload Storage   |
                                                        +----------------------+
```

## 4.2 各部分职责

### 客户端职责

- 显示主窗口和图表
- 响应用户操作
- 发起 API 请求
- 展示服务器返回的数据
- 做少量本地缓存和配置管理

客户端不再负责：

- 直接读写数据库
- 保持业务真相状态
- 直接从本地持久化会话

### 服务端职责

- 接收客户端请求
- 执行业务逻辑
- 管理数据集、会话、动作、订单线、统计
- 统一持久化
- 负责上传文件处理
- 负责审计日志、错误日志、备份策略

### 数据库职责

- 存数据集元信息
- 存 bars
- 存 sessions
- 存 actions
- 存 order lines
- 存 drawings
- 存统计结果或可重建状态

## 5. 与 Web 方案相比的优势

这条路线比“完整 Web 化”更适合你当前目标，原因如下。

## 5.1 最大化保留现有 UI

你现在最有价值、最难重做的，其实是桌面端的图表交互体验。

保留桌面客户端意味着：

- Qt 图表与交互可以继续用
- 不需要重写整套浏览器前端
- 现有操作习惯不必彻底改变

## 5.2 改造成本更低

相较于 Web 化，这条路线少了：

- 前端框架搭建
- 浏览器图表选型和重写
- Web 页面路由和状态管理
- 大量 UI 行为迁移

需要新增的是：

- 服务端 API
- 客户端 API 适配层
- 服务端部署和数据集中化

## 5.3 能更快得到你真正需要的收益

你的真实诉求是：

- 多台电脑共享同一份数据

而不是：

- 用浏览器访问

因此没有必要为“浏览器运行”付出重写前端的代价。

## 6. 核心设计原则

## 6.1 业务真相只存在于服务端

以后真正的数据源和业务状态应以服务端为准。

客户端可以有临时状态，但不能作为最终真相来源。

例如：

- 当前会话状态
- 当前仓位
- 动作列表
- 订单线状态
- 笔记和标签

都应由服务端持久化和返回。

## 6.2 客户端只做展示和交互，不做最终持久化

客户端的主要任务应变成：

- 发起请求
- 接收响应
- 更新 UI

不要让客户端继续保留大量“只有本地知道”的业务状态，否则以后很难同步。

## 6.3 核心逻辑保持单一实现

不论是桌面客户端还是将来可能出现的 Web 端，都应该共享同一套业务逻辑。

因此建议：

- 继续用 Python 保持核心领域逻辑
- 在服务端统一调用 `domain` 和 `storage`
- 客户端不要复制业务规则

## 7. 推荐的分层重构方案

建议将现有项目逐步调整为以下分层：

```text
src/barbybar/
  domain/
  data/
  storage/
  services/
  api/
  client/
  ui/
```

## 7.1 domain 层

保持纯业务性质：

- 不依赖 Qt
- 不依赖 FastAPI
- 不依赖 HTTP

它只处理：

- 回放推进
- 回退
- 跳转
- 交易动作影响
- 仓位变化
- 订单线触发
- 统计刷新

## 7.2 storage 层

短期职责：

- 提供数据库读写能力
- 支撑服务端的 Repository

长期建议：

- 避免 UI 直接访问 Repository
- 把 Repository 作为服务端内部组件

## 7.3 services 层

这是本次改造的关键新层。

建议提供明确的应用服务，例如：

- `DatasetService`
- `SessionService`
- `ReplayService`
- `OrderLineService`
- `DrawingService`

它们负责：

- 从 Repository 取数据
- 组装 `ReviewEngine`
- 执行业务动作
- 保存回数据库
- 返回 API 友好的结果

## 7.4 api 层

`FastAPI` 层负责：

- 定义路由
- 请求参数校验
- 响应格式标准化
- 错误处理
- 鉴权
- 文件上传

## 7.5 client 层

建议新增一个客户端 API 适配层，例如：

- `src/barbybar/client/api_client.py`

负责：

- 管理 `base_url`
- 发送 HTTP 请求
- 处理超时、错误码、重试
- 将 JSON 映射为客户端可用对象

## 8. 客户端如何演进

## 8.1 第一原则：UI 尽量少改，数据来源改掉

第一阶段不要急着大改桌面界面。

更稳妥的做法是：

- 尽量保留已有窗口结构
- 保留已有图表逻辑
- 把原来直连 `Repository` 的调用替换为 API 调用

### 当前模式

```text
UI -> Repository -> SQLite
```

### 目标模式

```text
UI -> ApiClient -> FastAPI -> Service -> Repository -> Database
```

## 8.2 客户端本地仍然会保留的内容

即使数据都放在服务器，客户端仍建议保留少量本地配置，例如：

- 服务器地址
- 登录 token
- 上次连接地址
- 窗口布局偏好
- 本地临时缓存
- 日志

## 8.3 客户端需要新增的能力

- 服务器连接配置
- 登录或授权
- 网络错误提示
- 请求超时处理
- 服务不可达提示
- 重新连接能力

## 9. 服务端设计建议

## 9.1 技术栈建议

推荐：

- Python 3.11+
- FastAPI
- Uvicorn / Gunicorn + Uvicorn Worker
- Pydantic

## 9.2 服务端建议提供的核心 API

### 连接与健康检查

- `GET /health`
- `GET /api/config`

### 认证

单人版可简化为：

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

### 数据集

- `GET /api/datasets`
- `POST /api/datasets/import`
- `GET /api/datasets/{id}`
- `DELETE /api/datasets/{id}`

### 会话

- `GET /api/sessions`
- `POST /api/sessions`
- `GET /api/sessions/{id}`
- `PATCH /api/sessions/{id}`
- `DELETE /api/sessions/{id}`

### 回放动作

- `POST /api/sessions/{id}/step-forward`
- `POST /api/sessions/{id}/step-back`
- `POST /api/sessions/{id}/jump`

### 交易动作

- `POST /api/sessions/{id}/actions`

### 订单线

- `GET /api/sessions/{id}/order-lines`
- `POST /api/sessions/{id}/order-lines`
- `PATCH /api/sessions/{id}/order-lines/{line_id}`
- `DELETE /api/sessions/{id}/order-lines/{line_id}`

### 绘图

- `GET /api/sessions/{id}/drawings`
- `PUT /api/sessions/{id}/drawings`

### 图表窗口

- `GET /api/sessions/{id}/chart-window`

## 9.3 响应设计建议

建议所有关键响应都带上：

- 当前 session 状态
- 当前 bar 位置信息
- 统计数据
- 订单线
- 必要时返回局部 chart 数据

## 10. 数据库选择建议

## 10.1 短期是否还能用 SQLite

可以，但要明确边界。

如果你只是：

- 单用户
- 同时只有一两个客户端开着
- 没有高并发写入

那服务端继续用 SQLite 是可行的。

## 10.2 更推荐的长期选择：PostgreSQL

如果你确定这个系统会长期用，甚至可能出现：

- 多台电脑同时开着
- 长期积累大量 bars 和 sessions
- 后续还会增加更多分析能力

更推荐尽早迁移到 PostgreSQL。

## 10.3 实际建议

更务实的选择是：

- 如果你想先快速验证，第一阶段服务端可以先用 SQLite
- 如果你准备长期部署，建议直接上 PostgreSQL

## 11. 文件上传与 CSV 导入设计

## 11.1 新模式建议

建议采用：

1. 客户端选择本地 CSV
2. 客户端上传文件到服务端
3. 服务端保存原始文件或临时文件
4. 服务端解析 CSV
5. 服务端写入数据库
6. 服务端返回导入结果

## 11.2 是否要保存原始 CSV

建议保存。

原因：

- 便于审计
- 便于重建数据集
- 便于排查解析问题
- 后续可支持重新导入

## 12. 会话状态管理设计

## 12.1 服务端思维

在客户端和服务端分离后，建议把“会话真相”放在服务端。

更具体地说：

- 每次重要操作都让服务端执行
- 服务端更新数据库
- 服务端返回新的 session 快照
- 客户端用返回值刷新界面

### 推荐的状态流转

```text
用户点击“下一根”
-> 客户端调用 step-forward API
-> 服务端加载 session + bars
-> 服务端执行 ReviewEngine.step_forward()
-> 服务端保存 session/actions/order_lines/stats
-> 服务端返回最新状态
-> 客户端刷新图表与面板
```

## 12.2 是否要在服务端保留长驻内存会话

第一阶段不建议。

建议采用“请求即加载，请求即保存”的无状态服务模式。

## 13. 通信策略

建议直接用：

- HTTPS + JSON API

没有必要一开始就引入：

- WebSocket
- gRPC
- GraphQL

## 14. 鉴权与安全建议

单人版也建议有最基本的保护。

推荐：

- 服务器只开放给内网或 Tailscale 网络
- 应用层再加一个简单账号密码或 token
- 公网访问时启用 HTTPS

## 15. 部署方案建议

## 15.1 推荐部署形态

推荐以下两种。

### 方案 1：Docker Compose

包含：

- `api` 服务
- `db` 服务
- `nginx`
- 数据卷

### 方案 2：系统进程直跑

包含：

- `FastAPI` 进程
- `PostgreSQL` 或 `SQLite`
- `Nginx`
- `systemd`

## 15.2 服务端目录建议

```text
/srv/barbybar/
  app/
  data/
    uploads/
    backups/
    logs/
  config/
```

## 16. 迁移路径建议

## 16.1 不建议的路径

不建议直接一步把桌面程序彻底改掉。

## 16.2 推荐路径

### Phase 0：确认目标与边界

- 单人版
- 保留桌面 UI
- 服务端统一存储
- 是否直接用 PostgreSQL
- 是否使用 Docker

### Phase 1：抽服务层

- 抽 `services`
- 统一用例入口
- 补关键测试

### Phase 2：搭建 FastAPI

- 定义 schema
- 定义 datasets/sessions/replay/actions 接口
- 实现健康检查与认证

### Phase 3：客户端增加 ApiClient

- 添加配置页
- 添加连接测试
- 封装 HTTP 请求

### Phase 4：逐步替换关键路径

按优先级替换：

1. 列表读取
2. 会话创建
3. 回放推进/回退
4. 动作记录
5. 订单线
6. 绘图
7. 导入

### Phase 5：切换到服务器模式为默认

- 客户端默认使用远程服务
- 本地直连模式可暂时保留

### Phase 6：清理与固化

- 清理旧路径
- 固化部署脚本
- 增加备份与日志方案

## 17. 关键难点

## 17.1 客户端状态与服务端状态一致性

解决原则：

- 操作以后，一律以服务端返回结果刷新界面

## 17.2 高频交互 API 设计

例如“下一根 K 线”这类操作会高频触发。

单人版初期建议先简单实现，不要过早优化。

## 17.3 图表数据量

建议：

- 服务端支持窗口查询
- 客户端只请求当前视图附近的 bars
- 跳转时按窗口重载

## 17.4 绘图和订单线同步

建议第一阶段先做到：

- 明确“编辑结束后提交”
- 不做每一帧实时同步

## 18. 工作量预估

## 18.1 最小可用版

目标：

- 服务端 API
- 客户端连接远程数据
- 数据集导入
- 会话管理
- 基础回放
- 动作记录

- 1 名熟悉当前代码的开发者

## 18.2 稳定长期自用版

新增：

- 鉴权
- PostgreSQL
- 备份
- 更完整错误处理
- 绘图同步
- 运维脚本

预估：

- 大约 3 到 6 周

## 19. 推荐的技术决策

如果让我给出当前最务实的一组建议，我会建议：

1. 保留桌面客户端
2. 后端用 FastAPI
3. 服务端数据库优先考虑 PostgreSQL
4. 客户端通过 HTTP API 访问服务端
5. 第一阶段采用无状态 API
6. 首先迁移数据集、会话、逐 bar 回放、动作记录
7. 绘图和复杂交互后置
8. 服务器通过 Docker Compose 或 Nginx + systemd 部署
9. 使用最基本的鉴权和 HTTPS

## 20. 最终建议

如果你的核心目标只是“多台电脑共享同一份个人数据”，那么没有必要优先做 Web 化。对当前项目来说，更优路线是：

- 保留桌面端
- 后端服务化
- 数据集中到服务器

一句话概括：

你要做的不是“网页化”，而是“把当前单机桌面工具升级成远程数据驱动的桌面客户端”。
